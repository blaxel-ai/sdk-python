# Blaxel Python SDK

Python SDK for the Blaxel platform. The main package is `blaxel.core`, with framework integrations in `blaxel.langgraph`, `blaxel.llamaindex`, `blaxel.openai`, `blaxel.crewai`, `blaxel.googleadk`, `blaxel.livekit`, `blaxel.pydantic`, and `blaxel.telemetry`.

## Repository structure

```
src/blaxel/
  core/
    client/                Auto-generated controlplane API client (api/, models/)
    sandbox/
      client/              Auto-generated sandbox API client (api/, models/)
      default/             Async sandbox subsystems
        drive.py           Drive mount/unmount operations
        filesystem.py      File read/write/search operations
        network.py         Port proxying (fetch)
        process.py         Process exec/logs/streaming
        codegen.py         Code editing (fastapply, reranking)
        preview.py         Preview management (control plane)
        session.py         Session management (control plane)
        sandbox.py         SandboxInstance - main entry point
        interpreter.py     CodeInterpreter (Jupyter)
        system.py          System upgrade
        action.py          Base class for all sandbox actions
      sync/                Synchronous versions of all sandbox subsystems
      types.py             Shared types (SandboxConfiguration, SessionWithToken, etc.)
    common/                Settings, logger, auth
    drive/                 DriveInstance (control plane CRUD)
    volume/                VolumeInstance (control plane CRUD)
    image/                 ImageInstance (image builds)
    jobs/                  JobInstance (job management)
    agents/                Agent helpers
    models/                Model gateway integrations
    tools/                 MCP tool helpers
    mcp/                   MCP server support
  langgraph/               LangGraph integration
  llamaindex/              LlamaIndex integration
  openai/                  OpenAI integration
  crewai/                  CrewAI integration
  googleadk/               Google ADK integration
  livekit/                 LiveKit integration
  pydantic/                Pydantic AI integration
  telemetry/               Telemetry support
tests/
  integration/core/sandbox/  Integration tests (requires BL_WORKSPACE + BL_API_KEY)
  integration/core/jobs/     Job integration tests
  integration/core/image/    Image integration tests
  integration/<framework>/   Framework-specific integration tests
```

## Code conventions

### Python
- Python 3.10+ with type hints throughout
- Uses `attrs` for data classes in generated code
- Uses `httpx` for HTTP clients (async and sync)
- snake_case for all naming (variables, functions, modules)
- No `print()` in library code -- use `logging` module

### Auto-generated code
- Files under `client/api/` and `client/models/` are auto-generated by `openapi-python-client`
- **Never edit generated files manually** -- they are overwritten on regeneration
- Regenerate with `make sdk-sandbox` or `make sdk-controlplane` (see `/regenerate-sdk` skill)
- Wrapper code (e.g. `drive.py`) should import and call functions from the generated `api/` modules, not make raw `httpx` calls

### Sandbox subsystem pattern
Every sandbox subsystem (async) follows this pattern:
```python
from ...common.settings import settings
from ..client.api.<module>.<endpoint> import asyncio as <function_name>
from ..client.client import Client
from ..client.models import ErrorResponse, SomeRequest, SomeResponse
from ..types import SandboxConfiguration
from .action import SandboxAction


class SandboxFoo(SandboxAction):
    def __init__(self, sandbox_config: SandboxConfiguration):
        super().__init__(sandbox_config)

    async def do_something(self) -> SomeResponse:
        client = Client(
            base_url=self.url,
            headers={**settings.headers, **self.sandbox_config.headers},
        )

        async with client:
            response = await some_function(client=client, body=request)
            if response is None:
                raise Exception("Failed to do something")
            if isinstance(response, ErrorResponse):
                raise Exception(f"Operation failed: {response.error}")
            return response
```

The sync version follows the same pattern with:
- `from .action import SyncSandboxAction`
- `with client:` instead of `async with client:`
- `sync` import instead of `asyncio` from the generated module

Key points:
- Extend `SandboxAction` (or `SyncSandboxAction`) for access to `self.url` and `self.sandbox_config`
- `self.url` respects `force_url` (for local dev / session-based access)
- Create a `Client` instance with `base_url` and merged headers
- Do NOT use raw `httpx.AsyncClient` directly or construct headers manually

### Exports
- All public types and classes must be re-exported through `sandbox/__init__.py`
- Async classes live in `sandbox/default/`, sync classes in `sandbox/sync/`
- Each sub-package has its own `__init__.py` with explicit `__all__`

### Testing
- Integration tests require `BL_WORKSPACE` and `BL_API_KEY` environment variables
- Use helpers from `tests/helpers.py` (`unique_name`, `default_labels`, `default_image`, etc.)
- Always label test resources with `default_labels` for automatic cleanup
- Always clean up sandboxes in class-level `cleanup` fixtures
- Run with: `uv run pytest tests/integration/core/sandbox/<file>.py -v -s`

### Linting and formatting
```bash
uv run ruff check --fix    # lint
uv run ruff format         # format
```

### Building
```bash
make build    # injects git commit hash into pyproject.toml
```

### SDK regeneration
```bash
make sdk                # both sandbox + controlplane
make sdk-sandbox        # sandbox only
make sdk-controlplane   # controlplane only
```
